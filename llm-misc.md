最近有个项目叫 openclaw 很火，然后有一个更精简版的叫做 nanoclaw。我其实对这些外围的项目了解非常有限，所以下载下来 nanoclaw，在 AI 的帮助下，基本复现了它的功能。

现在外围的项目如雨后春笋般出现，也昭示着大模型内核已经无法取得大幅度的进度了。我之前的想法是，我们一定要在大模型的召回率100%的那个前提下使用它，否则总会有隐患。实际这些外围的项目，设计的时候，大致使用了这个思路，但是没有很精确地规划这个问题，当然了，原因主要还是这些大模型公司本身不对外公开这个能力的阈值。不过我想，对于单个模型来说，测试一下也不费事吧，也许他们对外的单个模型内部也是使用了不同的服务，谁知道呢。

总而言之，大模型外围的项目普遍的做法是，对于一个确定的任务而言，先让大模型将任务切分成一步一步可执行的步骤。然后在单个步骤内，进行比如检验也好，或者人工的确认也好，尽量这个每一步做好，那么预期最后的结果就是好的。具体的切法依然是逻辑切割为主，而不是确保大模型的能力范围内。所以可以说是，尽力减少问题的概率，而不是避免问题。

nanoclaw 的接入层是一个聊天工具，例如 telegram，这是整个事件的驱动。收到消息之后，将消息发送给大模型，大模型在做这个任务的时候，有两类回复，一类是工具类回复，那么此时，需要调用更外围的例如 mcp 或者是 skills 的工具集。完成之后，又把结果发送给大模型，一直循环，直到大模型认为结束，或者是次数达到上限。还有一类就是语言回复，那和网页版的对话是一样的。

为了防止出现特别严重的问题，nanoclaw 沿用了之前的那个思路，减少问题而不是避免问题。即，让命令在 docker 的容器中执行。但这么做同时肯定也大幅度减少了这个工具的使用场景。

另外一个功能就是使用 SQLite 记录每次对话的 id 和回复，这样如果程序出问题，可以继续运行，因为大模型本身是没有记忆能力的。现有的做法是，将同一个任务的后续对话，拼接起来重复发送给大模型，这可真是一个浪费 token 的好办法。

明眼人都能看出来，这里缺少了一个本地的知识数据库。对于个人来说，这个数据库应该存储的是个人学识的一个备份，对于企业来说，应该是存储了一个业务的信息谱系。还好已经有人开始做了，我选择了 mem0 和 falkordb 这两个项目，希望可以了解一下这个领域。

很有意思的一点，因为大模型这个文本为基础的特性，又让古老的 unix 哲学焕发新生：

> 程序应该只关注一个目标，并尽可能把它做好。让程序能够互相协同工作。应该让程序处理文本数据流，因为这是一个通用的接口。
